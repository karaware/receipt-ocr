# MacBookを使わない原則無料OCR構成の検討

- 作成日: 2026-06-28
- ステータス: 検討案（未実装・未採用）
- 文書種別: 構成検討

## 目的

現在MacBookで実行している以下の処理を、常時起動できる無料枠のサーバーへ移す。

- Google Driveからの画像取得
- OCR
- レシート解析とカテゴリ分類
- Firestoreへの家計簿データ登録

運用上の前提は次のとおり。

- 通常の利用で費用を発生させない
- 1か月のレシート撮影は1,000枚より十分少ない
- OCRにはGoogle Cloud Vision APIの無料枠を使う
- Oracle Cloud Infrastructure（OCI）のAlways Free対象VMを使える
- MacBookの起動を通常運用の前提にしない
- Android専用アプリとGoogle Driveの既存アップロード経路は、可能な限り維持する

## 結論

推奨構成は、**Google Driveを画像の受け渡し場所として残し、OCI Always Free VMを常駐OCR
ワーカーにする**方式である。

```text
Android専用アプリ「レシート撮影」
  -> Google Drive / receipt-inbox
  -> OCI Always Free VMが定期取得
  -> Google Cloud Vision API / DOCUMENT_TEXT_DETECTION
  -> 既存Pythonロジックで解析・分類・整合性確認
  -> Cloud Firestore
  -> Firebase Hosting上の家計簿Webアプリ
```

この構成ではCloud Functions、Cloud Run、Cloud Storage for Firebaseを使わない。VMは外部から
HTTPリクエストを受けず、Drive、Vision、Firestoreへ外向き通信するだけにする。ドメイン、TLS証明書、
公開APIの認証を追加せずに済み、AndroidアプリのDriveアップロードも変更しなくてよい。

## なぜこの構成にするか

### AndroidからOCIへ直接送らない理由

OCIへ直接アップロードするには、公開HTTPS API、端末認証、TLS、レート制限、不正アップロード対策、
一時保存領域を新たに運用する必要がある。現在のDriveアップロードはGoogle OAuthとDriveの権限管理を
利用して動作しているため、まずはこれを維持する方が変更と攻撃面が小さい。

### Google Cloud Functionsを使わない理由

Cloud StorageへのアップロードをトリガーにCloud Functionsを動かす構成は技術的には自然だが、
FunctionsのデプロイにはFirebaseのBlazeプランが必要になる。今回の目的は無料運用であり、利用可能な
OCI Always Free VMがあるため、クラウド関数の役割をOCI VMへ移す。

### DriveをVMから定期確認する理由

Driveをイベント駆動で監視するとWebhookの公開、通知チャネルの更新、障害時の取りこぼし対策が必要になる。
家庭用の少量処理では、1～5分ごとのポーリングの方が単純で復旧しやすい。

## Google Cloudプロジェクトの分離

Cloud Vision APIは無料枠を使う場合でも、Google Cloudプロジェクトで請求を有効にする必要がある。
現在のFirebase家計簿はSparkプランと「請求先をリンクしない」方針で作られているため、同じプロジェクトへ
Vision APIの請求先を設定すると方針が崩れる。

次の2プロジェクトに分ける。

| プロジェクト | 用途 | 請求設定 |
| --- | --- | --- |
| Firebase家計簿プロジェクト | Authentication、Firestore、Hosting | Sparkのまま。請求先なし |
| OCR専用Google Cloudプロジェクト | Cloud Vision APIのみ | 請求先あり。無料枠内で使用 |

OCR専用プロジェクトではCompute Engine、Cloud Storage、Cloud Functions、Cloud Runを作らない。
Vision以外の課金対象リソースを置かないことで、請求対象を限定する。

## OCI VM上の処理フロー

VMではPythonワーカーをsystemd timerなどで1～5分ごとに起動する。

1. Drive APIで `receipt-inbox` のJPEGを列挙する。
2. DriveのファイルIDを使ってFirestoreの処理ジョブを確認する。
3. 未処理なら、当月のVision予約数が上限未満か確認する。
4. FirestoreトランザクションでVision利用枠を1件予約し、ジョブを `vision_reserved` にする。
5. 画像をVMの一時ディレクトリへダウンロードする。
6. Vision APIの `DOCUMENT_TEXT_DETECTION` を1回だけ呼ぶ。
7. OCRテキストを既存のparser、categorizer、reconciliationへ渡す。
8. レシートと明細をFirestoreへ冪等に書き込む。
9. 成功ジョブを `completed`、要確認を `needs_review`、失敗を `failed` にする。
10. 一時画像を削除する。
11. 必要ならDriveの元画像を `receipt-processed` へ移す。

VMが停止しても、元画像はDrive、処理状態はFirestoreに残る。VM内のSQLiteだけを唯一の処理履歴に
しないことで、Always Free VMを作り直しても処理を再開できる。

## Vision無料枠を超えない仕組み

Google Cloud公式料金表では、画像へ適用する機能ごとに1請求対象ユニットとなり、毎月最初の
1,000ユニットが無料である。1枚のJPEGに `DOCUMENT_TEXT_DETECTION` だけを適用する場合は、
原則として1画像が1ユニットになる。

ただし、請求先を有効にする以上、Google側に「料金が絶対に0円になる月次ハード上限」があるとは
みなさない。アプリ側で次の制御を行う。

- 自動処理の月次上限を **900ユニット** に固定し、100ユニットを安全余裕として残す
- `ocr_usage/{YYYY-MM}` をFirestoreトランザクションで更新する
- DriveファイルIDを一意キーにし、同じ画像へVisionを再実行しない
- Vision呼び出し直前に利用枠を予約する
- API送信後に応答不明となった場合は自動再試行せず、`unknown_after_request` として手動確認する
- `DOCUMENT_TEXT_DETECTION` 以外のVision機能を同じ画像へ追加しない
- OCR専用サービスアカウントをVM以外で使用しない
- Google Cloudの予算アラートとAPI使用量アラートを設定する
- Vision APIの分単位クォータも家庭用途に合わせて低くする

予算アラートは通知であり、利用や請求を自動停止する上限ではない。無料を保証する中心的な仕組みは、
公開APIを作らないこと、ファイルIDによる重複防止、VM側の900件上限である。

## 認証情報

VMには用途を分けた認証情報を置く。

| 認証情報 | 用途 | 権限方針 |
| --- | --- | --- |
| Drive用サービスアカウント | `receipt-inbox` の列挙・ダウンロード | 対象フォルダをサービスアカウントのメールへ共有。最初は読み取りだけ |
| Vision用サービスアカウント | OCR専用プロジェクトのVision API | Vision呼び出しに必要な最小権限だけ |
| Firebase Admin用サービスアカウント | Firestoreへの処理結果書き込み | 対象Firebaseプロジェクトだけ |

秘密鍵はGitへ保存せず、VM上の専用ディレクトリへ所有者だけ読める権限で配置する。例:

```text
/etc/receipt-ocr/secrets/
  drive-service-account.json
  vision-service-account.json
  firebase-service-account.json
```

```text
所有者: receipt-ocr
ディレクトリ: 0700
ファイル: 0600
```

サービスアカウント鍵が増えるため、実装時には1つのサービスアカウントへ安全に統合できるかも検証する。
統合する場合も、OCR専用プロジェクトの主体へFirebaseプロジェクトの必要最小限の権限だけを付与する。

## Drive側の扱い

最初の実装では次の順序で権限を増やす。

1. サービスアカウントへ `receipt-inbox` の閲覧権限だけを付与する。
2. VMから列挙とダウンロードができることを確認する。
3. 処理済み画像の移動が必要なら、`receipt-inbox` と `receipt-processed` の編集権限を付与する。
4. 移動権限の挙動を実ファイル1件で検証する。

サービスアカウントはDriveファイルを所有しない。Android利用者がアップロードしたファイルを共有権限で
読み取る構成にする。アーカイブ移動が不安定なら、Drive上の移動は行わず、FirestoreのDriveファイルIDで
処理済みを判定し、古い画像は家族アカウントから定期的に削除する。

## 既存コードの再利用と変更点

### そのまま再利用しやすいもの

- `src/receipt_ocr/parser.py`
- `src/receipt_ocr/categorizer.py`
- `src/receipt_ocr/reconciliation.py`
- `src/receipt_ocr/source_metadata.py`
- Androidアプリの撮影、画像連結、ファイル名生成、Driveアップロード
- Firebase Authentication、Firestoreルール、家計簿Webアプリ

### 変更が必要なもの

- `ocr.py` にGoogle Cloud Visionバックエンドを追加する
- Drive APIを直接使うサーバー用クライアントを追加する
- `pipeline.py` からローカルディレクトリとSQLiteへの強い依存を分離する
- `cloud.py` のSQLite行を前提とする同期処理を、解析結果から直接Firestoreへ書ける形にする
- Firestoreへ `ocr_jobs` と `ocr_usage` を追加する
- DriveファイルIDを基にした決定的なレシートIDを使う
- systemd service、timer、環境設定、ログローテーションを追加する
- Firestoreルールでは、処理ジョブと利用数を一般ユーザーから書き換えられないようにする

## Firestoreデータ案

```text
households/{householdId}/ocr_jobs/{driveFileId}
  status: discovered | vision_reserved | completed | needs_review | failed | unknown_after_request
  payer: me | wife
  driveFileId: string
  visionMonth: YYYY-MM
  visionUnits: 1
  error: string | null
  createdAt: timestamp
  updatedAt: timestamp

households/{householdId}/ocr_usage/{YYYY-MM}
  reservedUnits: number
  completedUnits: number
  limit: 900
```

既存の `receipts` と `transactions` はWebアプリとの互換性を保つ。OCR全文と画像本体はFirestoreへ
保存しない。要確認時はDriveファイルIDまたは家族だけがアクセスできるDriveリンクを保持する。

## 無料運用の見込み

| リソース | 使用方法 | 無料運用の条件・注意 |
| --- | --- | --- |
| OCI Compute VM | Pythonワーカーを常駐 | Always Free対象のホームリージョン・シェイプ・容量内で作成する |
| OCI Block Volume | OS、コード、短時間の一時ファイル | Always Free合計容量内に収める。画像を長期保存しない |
| Google Drive | Androidからの画像受け渡し | 家族アカウントの保存容量内。処理済み画像を定期整理する |
| Cloud Vision | `DOCUMENT_TEXT_DETECTION` 1機能だけ | 最初の1,000ユニット/月が無料。VM側は900件で停止する |
| Firestore | 家計簿、ジョブ、月次カウンタ | Spark無料枠内。家庭2人・月1,000件未満なら十分小さい想定 |
| Firebase Hosting | 静的Webアプリ | 無料の保存・転送枠内で使用する |
| Firebase Authentication | Googleログイン2人 | Sparkの無料利用範囲内 |
| Cloud Functions / Run / Storage | 使用しない | OCI VMで代替する |
| ドメイン・TLS | 使用しない | VMに公開HTTPエンドポイントを作らない |

## Oracle Always Freeの注意点

- Always Free対象リソースはテナンシのホームリージョンで作成する。
- Consoleで必ず **Always Free-eligible** と表示されるシェイプとストレージを選ぶ。
- リージョンで空きホストがない場合、Always Free VMを新規作成できないことがある。
- Oracleは低使用率のAlways Free Computeをアイドルと判断して回収する場合がある。
- 回収や障害を前提に、VM固有ディスクを正本にしない。
- VM再作成手順をスクリプト化し、DriveとFirestoreから処理を再開できるようにする。
- 無料枠を維持するためだけに無意味なCPU負荷を発生させない。
- OCI ConsoleでAlways Free対象外のシェイプ、追加ストレージ、バックアップを作らない。

Always Free VMは家庭用途には十分だが、SLAのある常時稼働サービスとしては扱わない。停止中はDriveに
画像がたまるだけにし、VM復旧後に順次処理する。

## セキュリティ

- VMの受信ポートは原則SSHだけにする
- SSHは公開鍵認証のみ、パスワードログインとrootログインを無効にする
- 可能なら接続元IP制限またはOCI Bastionを使う
- OSセキュリティ更新を自動適用する
- ワーカーはrootではなく専用OSユーザーで動かす
- 一時画像は処理後に必ず削除する
- ログへOCR全文、カード番号、サービスアカウント鍵を出さない
- サービスアカウント鍵を定期的にローテーションする
- Vision用Google Cloudプロジェクトでは不要なAPIを有効にしない

## 障害時の動き

| 障害 | 動作 |
| --- | --- |
| OCI VM停止 | Driveに画像が残る。復旧後に再開 |
| Drive API失敗 | Visionを呼ばず次回へ延期 |
| Vision API失敗（送信前と確定） | 予約状態を戻して限定回数だけ再試行 |
| Vision API応答不明 | 再課金防止のため自動再試行せず手動確認 |
| OCR解析失敗 | `needs_review` としてFirestoreへ記録 |
| Firestore書き込み失敗 | 同じ決定的IDで再試行し、重複を防止 |
| 月900件到達 | OCRを停止し、管理者へ通知。翌月まで自動再開しない |

## 実装順序

### フェーズ1: 小さな実証

1. OCR専用Google Cloudプロジェクトを作成する。
2. 請求先、Vision API、予算アラートを設定する。
3. OCI Always Free VMを作成する。
4. VMからテスト画像1枚を `DOCUMENT_TEXT_DETECTION` へ送り、結果を保存する。
5. Driveの対象フォルダをサービスアカウントへ共有し、VMから1枚ダウンロードする。
6. Firestore Emulatorまたはテスト用コレクションへ解析結果を書き込む。

### フェーズ2: ワーカー実装

1. OCR、解析、Firestore書き込みをSQLiteから分離する。
2. `ocr_jobs`、`ocr_usage`、900件上限を実装する。
3. systemd serviceとtimerを追加する。
4. ログ、失敗状態、手動再実行コマンドを実装する。

### フェーズ3: 移行

1. 家族のテストレシートでAndroidからFirestoreまで通す。
2. 同じ画像を再検出してもVisionを再実行しないことを確認する。
3. VM停止・再作成後に処理を再開できることを確認する。
4. 1～2週間、Apple Visionの結果と比較する。
5. 問題がなければMacの定期OCR運用を停止する。
6. Mac版は緊急時の手動フォールバックとして残す。

## 必須テスト

- 同一Driveファイルを何度列挙してもVision呼び出しが1回だけになる
- 月次予約数が900件を超えない
- 月が変わると新しいカウンタへ切り替わる
- Visionのタイムアウトで無条件再試行しない
- VM再起動後に未完了ジョブから再開できる
- Firestoreの同一レシート・明細を重複作成しない
- 支払者名をAndroidのファイル名から復元できる
- OCR失敗や不足項目を `needs_review` としてWebへ表示できる
- サービスアカウントに対象外のDriveファイルが見えない

## 採用判断

この構成は次の条件を満たすため、実証へ進める価値がある。

- MacBookを起動しなくてよい
- Androidアプリの既存Driveアップロードを維持できる
- Cloud Functionsなどの有料化につながる実行基盤を使わない
- Vision以外は現在の無料枠とOCI Always Freeで構成できる
- Visionは通常利用で月1,000ユニット未満、実装上は900件で停止できる

ただし、Vision APIには請求先の有効化が必要であり、Oracle Always Free VMには回収・容量不足の可能性が
ある。そのため「原則無料」は実現可能だが、「どのような障害や設定ミスがあっても絶対に0円」とは
保証しない。OCR専用プロジェクトの分離、900件上限、重複防止、最小権限、利用量監視を採用条件とする。
