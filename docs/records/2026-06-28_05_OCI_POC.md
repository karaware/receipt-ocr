# OCI VMを使ったCloud Vision OCRのPoC計画

- 作成日: 2026-06-28
- ステータス: PoC案（未実施）
- 文書種別: 実証計画
- 関連: [MacBookを使わない原則無料OCR構成の検討](2026-06-28_03_CONSIDER.md)
- 関連: [Cloudflare無料機能の検討](2026-06-28_04_cloudflare_consider.md)

## 結論

PoCは、最初からAndroid、Drive、Vision、Firestore、Webをすべて変更するのではなく、次の順番で進める。

```text
1. OCI VMからCloud Visionを1回呼ぶ
2. OCI VMからDriveの画像を1枚取得する
3. 既存PythonでOCR結果を解析する
4. PoC用Firestoreコレクションへ書く
5. 同じ画像を二重処理しないことを確認する
6. systemd timerで5分ごとに自動処理する
7. 最後にAndroidからのアップロードを通す
```

PoC中のVision呼び出しは最大20ユニットに固定する。本番用FirestoreデータとDriveフォルダは使わず、
PoC専用の保存先へ分離する。

## PoCで確認すること

主な不確実性は次の5点。

1. OCI Always Free VMを安定して作成・再起動できるか
2. サービスアカウントから家族のDriveフォルダを列挙・ダウンロードできるか
3. OCI VMからCloud Vision APIを無料枠内で呼べるか
4. 既存のparser、categorizer、reconciliationをLinux上で再利用できるか
5. Visionの重複呼び出しを防ぎながらFirestoreへ冪等に登録できるか

これらが確認できれば、MacBookを通常運用から外す構成は実現可能と判断する。

## PoCで確認しないこと

最初のPoCでは以下を実装しない。

- Androidアプリのアップロード先変更
- Cloudflare Workers、R2、Queues
- Google DriveからOCIへのイベント通知
- Drive画像の自動削除
- 本番Firestoreの `receipts` / `transactions` への登録
- Web画面の新機能
- 高可用性、複数VM、ロードバランサ
- OCI VM上の公開HTTP API
- OCR精度の全面的な改善

PoCは「技術的に一連の処理が動くか」と「無料運用を妨げる問題がないか」の確認に限定する。

## PoC専用リソース

本番データと混ざらないよう、次を用意する。

| リソース | PoC用の値 |
| --- | --- |
| Drive入力フォルダ | `receipt-inbox-poc` |
| Drive処理済みフォルダ | PoCでは作らない。元画像を残す |
| Google Cloudプロジェクト | OCR専用プロジェクト |
| Vision機能 | `DOCUMENT_TEXT_DETECTION` だけ |
| Firestoreジョブ | `households/{householdId}/poc_ocr_jobs/{driveFileId}` |
| Firestore結果 | `households/{householdId}/poc_receipts/{driveFileId}` |
| 月次使用数 | `households/{householdId}/poc_ocr_usage/{YYYY-MM}` |
| Vision上限 | PoC全体で20ユニット |
| VM上の一時画像 | `/var/lib/receipt-ocr-poc/work/` |
| ログ | systemd journal。OCR全文は出さない |

PoCのFirestoreコレクションは、終了後にまとめて削除できる名前にする。

## フェーズ0: 課金と秘密情報のガードを先に作る

Vision APIを有効にする前に以下を行う。

1. Firebase家計簿とは別のOCR専用Google Cloudプロジェクトを作る。
2. OCR専用プロジェクトだけに請求先をリンクする。
3. Cloud Vision API以外の課金対象サービスを作らない。
4. 少額の予算アラートを設定する。
5. Vision APIの分単位クォータを家庭用途に合わせて低くする。
6. PoCコードに `POC_MAX_VISION_UNITS=20` を設定する。
7. 利用数が20に達したら、例外ではなく安全停止する。

予算アラートは支出を停止する上限ではない。20ユニット制限、重複防止、サービスアカウント鍵を公開しない
ことを主な防止策とする。

## フェーズ1: OCI Always Free VMを準備する

### VM条件

- ホームリージョンに作成する
- OCI Consoleで **Always Free-eligible** と表示されるシェイプを選ぶ
- Always Free対象のBoot Volume容量内に収める
- UbuntuまたはOracle Linuxを使用する
- 外部公開する受信ポートはSSHだけにする
- SSHは公開鍵認証のみを使う
- パスワードログインとrootログインを無効にする

ArmとAMDのどちらでもよい。Python、Firebase Admin SDK、Google APIクライアントがインストールできる方を
選ぶ。Armで依存関係に問題が出た場合は、PoC結果として記録し、Always Free対象の別シェイプを試す。

### VMへ入れるもの

```text
git
python3
python3-venv
ca-certificates
receipt-ocrリポジトリ
Python仮想環境
```

Dockerは最初のPoCでは使わない。systemdとPython venvの方が構成が少なく、VMのメモリも節約できる。

### フェーズ1完了条件

- VMを停止・起動してもSSH接続できる
- Pythonで既存ユニットテストを実行できる
- Google APIのHTTPSエンドポイントへ外向き接続できる
- OCI Console上でAlways Free対象外のリソースがない

## フェーズ2: Vision API単体テスト

Drive連携より先に、VMからVisionだけを呼ぶ。

1. Vision専用サービスアカウントを作る。
2. Vision呼び出しに必要な最小権限を付ける。
3. JSON鍵をVMの `/etc/receipt-ocr-poc/secrets/` へ置く。
4. 所有者を専用OSユーザー、ディレクトリを `0700`、鍵を `0600` にする。
5. リポジトリのテスト用レシート画像1枚をVMへ置く。
6. `DOCUMENT_TEXT_DETECTION` を1回呼ぶ。
7. OCR結果を一時ファイルへ保存し、日本語が取得できることを確認する。
8. Google Cloud ConsoleでAPI使用数を確認する。

### フェーズ2完了条件

- VisionがHTTPエラーなしで結果を返す
- OCR結果に日本語文字列が含まれる
- 呼び出した機能が `DOCUMENT_TEXT_DETECTION` 1つだけ
- PoC利用数カウンタが1になる
- 意図しないGoogle Cloudリソースや費用が発生していない

## フェーズ3: Drive API単体テスト

Visionとは別に、VMからDriveのPoCフォルダを読めるか確認する。

1. Google Driveに `receipt-inbox-poc` を作る。
2. テスト用JPEGを1枚だけ入れる。
3. Drive用サービスアカウントのメールアドレスへフォルダを閲覧者として共有する。
4. VMからフォルダIDを指定してJPEGを列挙する。
5. ファイルID、ファイル名、MIME type、サイズを取得する。
6. 画像を一時ディレクトリへダウンロードする。
7. SHA-256を計算し、Drive上の元画像と同じ内容であることを確認する。

PoCではDriveの更新、移動、削除を行わない。最初は読み取り専用権限で十分。

### フェーズ3完了条件

- サービスアカウントがPoCフォルダ内のファイルだけを取得できる
- 対象外の家族Driveファイルを列挙できない
- JPEGを破損せずダウンロードできる
- 同じファイルのDriveファイルIDが毎回変わらない

### 失敗した場合

サービスアカウントへのフォルダ共有で安定して取得できなければ、次の順で検討する。

1. 共有権限とフォルダIDを再確認する
2. Drive用サービスアカウントとVision用サービスアカウントを分離する
3. 家族共用GoogleアカウントのOAuth refresh tokenをVMへ安全に保存する方式を検討する

3は秘密情報とトークン更新の運用が増えるため、最後の選択肢とする。

## フェーズ4: 既存Python解析のLinux確認

保存したVision OCRテキストを既存コードへ渡す。

```text
Vision OCRテキスト
  -> parser.parse_receipt
  -> categorizer.categorize_items
  -> reconciliation.reconcile_receipt
```

このフェーズではApple Visionとの精度比較ではなく、Linux上で既存ロジックが動くことを確認する。

### テストデータ

最初は5枚程度を使う。

- スーパー
- コンビニ
- ドラッグストア
- 外食
- 長いレシート

### 確認項目

- 店名
- 日付
- 合計金額
- 明細候補
- 支払者名
- `confirmed` または `needs_review` の判定

### フェーズ4完了条件

- 例外でワーカー全体が停止しない
- 取得できない項目は空欄または `needs_review` になる
- 少なくとも店名、日付、合計を既存データ構造へ変換できる
- Android専用アプリ形式のファイル名から支払者名を復元できる

## フェーズ5: FirestoreへのPoC登録

本番の `receipts` と `transactions` へは書かず、PoC専用コレクションへ登録する。

### ジョブ例

```text
households/{householdId}/poc_ocr_jobs/{driveFileId}
  status: discovered | vision_reserved | completed | needs_review | failed | unknown_after_request
  driveFileId: string
  sourceName: string
  payer: string
  visionUnits: 1
  error: string | null
  createdAt: timestamp
  updatedAt: timestamp
```

### 結果例

```text
households/{householdId}/poc_receipts/{driveFileId}
  shopName: string
  purchasedAt: string | null
  totalAmount: number | null
  payer: string
  status: confirmed | needs_review
  reviewReason: string | null
```

DriveファイルIDをFirestoreドキュメントIDとして使う。同じファイルを再検出した場合、既存ジョブが
`completed` または `needs_review` ならVisionを呼ばない。

### Vision呼び出し前の状態更新

1. Firestoreトランザクションを開始する。
2. PoC全体の予約数が20未満か確認する。
3. 未処理ファイルなら予約数を1増やす。
4. ジョブを `vision_reserved` にする。
5. トランザクション成功後にVisionを呼ぶ。

Visionへの送信後に応答が不明になった場合は `unknown_after_request` とし、自動再試行しない。

### フェーズ5完了条件

- 同じDriveファイルを10回列挙してもVision呼び出しは1回だけ
- Firestore結果ドキュメントも1件だけ
- VM再起動後も処理済み判定を維持できる
- Firestoreルール上、一般Web利用者がPoC利用数を変更できない
- OCR全文や画像本体をFirestoreへ保存しない

## フェーズ6: 1回だけ動くワーカーを作る

最初は常駐プロセスではなく、起動すると対象を処理して終了するCLIにする。

```bash
python -m receipt_ocr cloud-worker --once --poc
```

想定する処理:

```text
Drive列挙
  -> 未処理を最大1件選択
  -> 利用数予約
  -> ダウンロード
  -> Vision
  -> 解析
  -> Firestore
  -> 一時画像削除
  -> 終了
```

1回の起動で最大1件に限定すると、障害箇所、ログ、Vision使用数を確認しやすい。

### 必要なCLI

```bash
# 未処理ファイルを表示するだけ
python -m receipt_ocr cloud-worker --poc --dry-run

# 1件処理
python -m receipt_ocr cloud-worker --poc --once

# ジョブ状態確認
python -m receipt_ocr cloud-worker status --poc

# unknown状態を人が確認後に再実行可能にする
python -m receipt_ocr cloud-worker retry <drive-file-id> --poc
```

## フェーズ7: systemd timerで自動化する

CLIが安定してから、5分ごとに起動する。

```text
receipt-ocr-poc.service
receipt-ocr-poc.timer
```

serviceは1回処理して終了し、timerが次回起動する。常駐ループよりも、失敗時の再起動、ログ確認、
デプロイが単純になる。

### 確認すること

- 同時に2プロセス起動しない
- 前回処理中なら次回を重ねない
- Drive API失敗時にVisionを呼ばない
- 失敗しても次回timerが動く
- `journalctl` でファイルIDと状態を追跡できる
- ログにOCR全文、カード番号、秘密鍵を出さない

### フェーズ7完了条件

- 24時間以上、自動実行が継続する
- VM再起動後にtimerが自動復旧する
- 画像追加から10分以内にFirestoreへ結果が入る
- 重複処理がない

## フェーズ8: Androidからのエンドツーエンド確認

最後にAndroidアプリからPoCフォルダへ送る。

1. Androidアプリの保存先を一時的に `receipt-inbox-poc` へ変更する。
2. 支払者名を設定した状態でレシートを1枚撮る。
3. Driveへのアップロードを確認する。
4. OCI workerが自動取得するまで待つ。
5. FirestoreのPoC結果を確認する。
6. ファイル名から支払者が復元されていることを確認する。
7. Androidアプリの保存先を元に戻す。

本番フォルダを監視対象に追加するのは、PoC合格後にする。

## テスト枚数

PoC全体で最大20回に収める。

| 用途 | 最大Vision呼び出し数 |
| --- | ---: |
| Vision単体確認 | 1 |
| 既存parser確認 | 5 |
| Firestore・冪等性確認 | 5 |
| systemd自動実行 | 5 |
| Android E2E | 2 |
| 予備 | 2 |
| 合計 | 20 |

同じOCRレスポンスを保存して解析テストへ再利用し、解析コードのテストではVisionを再度呼ばない。

## PoC合格条件

以下をすべて満たしたら、本番実装へ進む。

- OCIリソースがAlways Free対象内
- Google Cloudの請求額が0円
- Vision呼び出しがPoC上限20以下
- Driveから安定して画像を取得できる
- 同じDriveファイルに対するVision呼び出しが1回だけ
- VM再起動後も処理済み状態を維持できる
- 画像追加から10分以内にFirestoreへ結果が入る
- 店名、日付、合計の不足を `needs_review` として扱える
- Firestoreへレシートや明細を重複登録しない
- OCR全文、画像、鍵をログやFirestoreへ不要に残さない
- 24時間の自動実行で停止しない

OCR精度はPoC合否と分けて記録する。精度が不十分でも、パイプラインが安定し要確認へ回せるなら、
基盤のPoCは合格とする。

## PoC中止条件

次の場合はPoCを止め、原因と代替案を記録する。

- Always Free対象VMを確保できない
- OCIで意図しない課金対象リソースが作られる
- Vision以外のGoogle Cloud費用が発生する
- サービスアカウントでDriveを安全に限定公開できない
- 同じ画像を重複OCRする問題を解消できない
- Firestoreの本番データとPoCデータを分離できない
- 秘密鍵を安全に保管できない

代替候補はCloudflare Workerの最小PoC、または家族共用GoogleアカウントのOAuthを使うOCI workerとする。

## PoC後に本番化するときの変更

PoC合格後に次を行う。

- `poc_ocr_jobs` を本番 `ocr_jobs` へ変更する
- `poc_receipts` ではなく既存 `receipts` / `transactions` へ書く
- 月次上限を20から900へ変更する
- 本番 `receipt-inbox` を監視する
- 必要なら `receipt-processed` への移動権限を追加する
- Web画面へ `needs_review` とワーカー最終実行時刻を表示する
- OCI VM再構築手順とバックアップ不要な設計を文書化する
- MacBookの定期OCRを停止する
- Mac版は手動フォールバックとして残す
