# ADR-0002: OCRワーカーの実行基盤にOCI Computeを採用する

- ステータス: 採用
- 記録日: 2026-06-28
- 決定日: 2026-06-28

## コンテキスト

レシート画像は専用AndroidアプリからGoogle Driveへアップロードし、PythonのOCRパイプラインで解析して
Firestoreへ保存する。これまでMacBook上でOCR処理を動かしていたが、MacBookの起動状態、Google Drive for
desktopの同期状態、手動CLI実行に依存するため、画像追加から家計簿反映までを常時自動化できない。

既存のPythonコードには、Drive画像取得、Cloud Vision OCR、レシート解析、カテゴリ分類、差額照合、
Firestoreへの冪等登録がある。新しい実行基盤でもこれらをできるだけ変更せず再利用したい。

家庭用途のため、次の条件を重視する。

- 原則無料で常時稼働できる
- PythonとGoogle Cloud SDKをそのまま実行できる
- 5分間隔の少量バッチを安定して処理できる
- SSH以外の受信ポートを公開しない
- OCR画像と認証鍵を恒久的に外部公開しない
- Vision呼び出しの重複と意図しない課金を防げる

## 決定

OCRワーカーの実行基盤として、Oracle Cloud InfrastructureのCompute VMを採用する。

初期構成は次のとおり。

| 項目 | 採用値 |
| --- | --- |
| リージョン | テナンシのホームリージョンであるIndia West (Mumbai) |
| シェイプ | Always Free対象の `VM.Standard.A1.Flex` |
| リソース | 1 OCPU、6GBメモリ |
| OS | 通常版Oracle Linux 9 ARM64 |
| ブートボリューム | デフォルト50GB相当、Balanced、Oracle管理キーで暗号化 |
| ネットワーク | Public IPv4を持つ公開サブネット |
| Ingress | 管理元グローバルIPv4 `/32` からのTCP 22だけ |
| 実行方式 | Python venvとsystemd oneshot service/timer |
| 実行間隔 | 5分ごと、1回につき最大1画像 |

VMはGoogle Driveの専用フォルダを読み取り、未処理画像だけをCloud Visionへ送り、既存のparser、categorizer、
reconciliationを通してFirestoreへ書き込む。同じDriveファイルIDに対するジョブ状態をFirestoreへ保持し、
Vision呼び出しとレシート登録の重複を防ぐ。

OCIはOCRワーカーだけを担当する。以下は既存構成を維持する。

- Google Drive: Androidからのレシート画像入力
- Cloud Vision: OCR
- Firebase Authentication: 本人・妻のGoogleログイン
- Firestore: 家計簿データの正本
- Firebase Hosting: 家計簿Webアプリ

PoCでは専用Driveフォルダ、専用Firestoreコレクション、Vision最大20ユニットを使う。PoC合格後に本番
コレクションと本番Driveフォルダへ切り替え、MacBookの定期OCRを停止する。MacBook処理は障害時の手動
フォールバックとして残す。

## 採用理由

- Always Freeの範囲で、家庭用途に十分なCPU、メモリ、ブートボリュームを利用できる
- ARM64上で既存のPythonとGoogle APIクライアントを実行できる
- Cloudflare Workers向けにPython処理を再設計する必要がない
- systemd timerにより、常駐アプリや公開HTTP APIなしで定期処理できる
- VM上の一時ファイル、サービスアカウント権限、OSユーザー権限を明示的に制御できる
- FirebaseとGoogle Driveの既存データフローを維持できる

## 検討した代替案

### MacBookでの継続運用

既存実装をそのまま利用できるが、MacBookの起動、同期、手動実行に依存し、家計簿反映を常時自動化できない。
フォールバックとしては残すが、主実行基盤にはしない。

### Cloudflare Workers

無料枠とエッジ実行は魅力的だが、既存Python処理、Google認証、Visionレスポンス解析をWorkersの制約内へ
移植する必要がある。CPU時間制限と実装変更量がPoCの目的に合わないため、主ワーカーには採用しない。

### Google Cloud RunまたはCloud Functions

Google APIとの統合は容易だが、無料運用の把握、請求先設定、実行時間・呼び出し回数の管理対象が増える。
Cloud Vision以外の実行費を持たない構成を優先し、今回は採用しない。

### Firebaseだけで処理する

Firestore、Authentication、Hostingには適するが、Firebase HostingはPython OCRワーカーを実行しない。
Functionsを追加するとGoogle Cloudの実行基盤と課金管理が必要になるため、OCR実行先にはしない。

## 結果

### 利点

- MacBookが停止していてもDriveからFirestoreまで自動処理できる
- 既存Pythonコードを大きく変更せず利用できる
- 画像処理を1件ずつ直列化し、Vision利用数と障害箇所を追跡できる
- VMを再作成してもcloud-initで同じ環境を構築できる
- 公開サービスを持たず、SSHの接続元を限定できる

### コストと制約

- Always Freeの容量、シェイプ、ホームリージョン条件を継続的に確認する必要がある
- Mumbaiリージョンのため、日本からのSSH操作には遅延がある
- VMのOS更新、鍵ローテーション、systemd監視を管理する必要がある
- Always Freeのホスト容量不足やOracle側の条件変更でVMを確保できない可能性がある
- Cloud VisionはOCI無料枠とは別であり、利用上限とGoogle Cloudの予算監視が必要
- Public IPv4を持つため、SSHのIngress `/32` と公開鍵認証を維持する必要がある

## 運用上のガード

- cloud-initへ秘密情報を埋め込まない
- サービスアカウント鍵は `/etc/receipt-ocr-poc/secrets/` に`0600`で配置する
- Drive、Vision、Firestoreのサービスアカウントを分離し、最小権限にする
- OCR全文、画像、鍵をsystemd journalやFirestoreへ保存しない
- PoCのVision予約数を20で停止する
- `unknown_after_request` は自動再試行しない
- systemd timerは設定とdry-run確認が完了するまで有効化しない

## 採用を継続する条件

次を確認できた場合、OCI VMを本番ワーカーとして継続使用する。

- cloud-initが正常終了し、VM再起動後も環境が維持される
- Drive、Vision、Firestoreの単体接続が成功する
- 同じDriveファイルを再検出してもVisionを再呼び出ししない
- 画像追加から10分以内にFirestoreへ結果が入る
- 24時間のtimer運転で停止・重複処理がない
- OCIとGoogle Cloudに意図しない費用が発生しない

満たさない場合は、MacBook運用へ戻した上で、Google Cloudのマネージド実行基盤またはCloudflare構成を
再評価する。
