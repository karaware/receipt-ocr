# ADR-0003: OCRワーカーを5分ごとの最大4件バッチと月800件上限で運用する

- ステータス: 採用
- 記録日: 2026-07-20
- 決定日: 2026-07-20

## コンテキスト

OCRワーカーは5分ごとのsystemd timerで起動する。当初は1回につき1画像、Vision利用上限はPoC全体で20件だった。
複数のレシートを連続撮影すると、処理待ちが5分ごとに1件ずつしか進まず、確認待ちへの反映に時間がかかった。

また、Cloud Visionの無料利用枠は月単位である。累計20件で止める制限では、無料枠を使い切る前に処理が停止する。
家庭用途では、無料範囲を保ちながら、通常のまとめ撮影を待たせない運用が必要である。

## 決定

- systemd timerは5分間隔を維持する。
- 1回の起動で、完了済みLLM結果の反映と新規Drive画像の処理をそれぞれ最大4件まで行う。
- 件数は `poc.max_images_per_run` または `POC_MAX_IMAGES_PER_RUN` で設定し、既定値を4とする。
- Cloud Visionの新規OCR予約は月800画像で停止する。
- 月別使用量はFirestoreの `households/{householdId}/poc_ocr_usage/YYYY-MM` の `units` で管理する。累計値 `_total` は監視用にのみ保持し、上限判定には使わない。
- 上限到達時は新規画像を `limit_reached` として保留し、翌月の最初の実行から自動的に処理対象へ戻す。

## 採用理由

- 4件なら、20件の確認待ちをおおむね25分でOCRキューへ投入でき、1件ずつの100分より実用的である。
- Cloud Visionの無料枠は月ごとにリセットされるため、上限も月単位にするのが費用管理として正しい。
- Firestoreのトランザクションで予約と月別カウントを一体化し、同時実行時にも上限超過を防げる。
- 上限未満の通常運用では、利用者による手動解除が不要である。

## 検討した代替案

### 1回1件のまま維持する

実装は単純だが、まとめ撮影時の待ち時間が長く、実機テストで不便さが明確になったため採用しない。

### Vision上限を累計20件のまま維持する

PoCの安全確認には適するが、月単位の無料枠を正しく利用できず、実運用には採用しない。

### 上限を設けない

意図しない大量アップロードで課金リスクが生じるため採用しない。

## 結果

### 利点

- 連続撮影した複数レシートが1回のtimer実行で進む
- 無料利用の範囲を明確な月800件に保てる
- `limit_reached` の翌月復旧が自動化される

### コストと制約

- 月800件を超える画像は翌月までOCRされない
- 1回最大4件のため、大量のバックログは複数回のtimer実行を要する
- 上限値はGoogle Cloudの契約・料金体系変更時に見直す必要がある
